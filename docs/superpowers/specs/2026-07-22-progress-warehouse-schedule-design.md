# BL 진행현황 반입 일정 편집 및 알림 설계

## 목표

`BL별 진행현황`에서 반입(예정)구역과 반입예정일을 확인·수정하고, 화주가 값을 실제 변경한 경우 지정된 내부 담당자에게 변경 메일을 발송한다. 중복되는 상단 `원본필수서류 수취현황` 메뉴는 제거하고 반입예정일을 진행현황 달력에도 표시한다.

## 범위

- 상단 도구모음에서 `원본필수서류 수취현황` 버튼만 제거한다.
- 기존 원본서류 데이터, 카드 내 O/X 표시, 모바일 원본서류 페이지와 관련 API는 유지한다.
- `BL별 진행현황`의 `반입(예정)구역` 오른쪽에 `반입예정일` 열을 추가한다.
- 진행현황 표에서 반입구역 또는 반입예정일을 클릭하면 두 값을 함께 편집하는 소형 모달을 연다.
- 반입구역은 기존 `storage_yard`, 반입예정일은 기존 `warehouse_expected_date`를 사용한다.
- 반입 일정 편집을 위해 별도 Supabase 컬럼이나 별도 API 엔드포인트를 만들지 않는다.

반입 일정 기능 자체는 기존 컬럼과 기존 API만 사용한다. 모바일 양도증 수동 상태를 로컬 스캔 결과와 분리해 보존하기 위해서는 `cargo_original_docs.transfer_received_override boolean` 컬럼 하나를 추가한다.

## 저장 및 메일 흐름

1. 사용자가 진행현황 표에서 반입 정보를 편집하고 저장한다.
2. 브라우저는 기존 `/api/cargo-quota`의 `manual_fields` 작업으로 전체 수기 운송정보를 전송한다.
3. API는 기존 사용자 입력값을 먼저 조회한 뒤 저장 요청값과 비교한다.
4. 변경값을 기존 `cargo_card_user_inputs` 행에 upsert한다.
5. 로그인 역할이 `shipper`이고 `storage_yard` 또는 `warehouse_expected_date`가 실제 변경된 경우에만 메일 한 통을 발송한다.
6. 관리자 역할의 수정과 동일 값 재저장은 메일을 발송하지 않는다.
7. 저장은 성공했지만 SMTP 발송이 실패하면 저장 결과를 유지하고 응답에 메일 실패 사유를 포함한다.

## 메일

수신자는 다음 주소로 고정한다.

- `jsh@aincustoms.com`
- `jhcho@aincustoms.com`
- `bill@aincustoms.com`
- `ain@aincustoms.com`

제목은 `[반입정보 변경] 화주명 / B/L` 형식으로 한다. 본문에는 변경자 계정, 화주명, B/L, 반입구역의 변경 전·후 값, 반입예정일의 변경 전·후 값, 변경 시각을 포함한다. 변경되지 않은 항목도 현재 상태 확인을 위해 함께 표시하되, 실제 변경 항목을 별도로 명시한다.

## 화면 동작

- 진행현황 표에는 `반입(예정)구역`, `반입예정일` 두 열을 연속 배치한다.
- 각 값은 표의 밀도를 해치지 않는 작은 편집형 버튼으로 표시한다.
- 두 열 중 어느 쪽을 눌러도 같은 편집 모달이 열린다.
- 저장 성공 시 전체 대시보드를 다시 그리지 않고 현재 카드 데이터와 진행현황 행만 갱신한다.
- 메일 실패 시 `입력값은 저장되었지만 알림 메일 발송에 실패했습니다`라는 경고를 표시한다.

## 달력

`BL별 진행현황` 하단 달력의 기존 일정 생성 함수에 `warehouse_expected_date`를 추가한다. 날짜가 있는 B/L은 `반입예정 · B/L · 반입구역` 일정으로 표시하며, 기존 입항예정일과 원본서류 수령요청일 일정은 그대로 유지한다.

## 원본서류 토글

- `BL별 진행현황`의 BL원본 및 H/C원본이 X인 상태에서 클릭하면 기존처럼 수령일 입력창을 열고, 유효한 수령일 입력 후 O로 저장한다.
- O인 상태에서 클릭하면 수령일 입력창을 열지 않고 `O를 X로 변경할까요?` 확인창을 연다.
- 확인하면 선택한 원본 상태만 X로 변경한다. 다른 원본이 O로 남아 있으면 실제수령일은 유지하고, BL원본과 H/C원본이 모두 X가 되면 실제수령일을 비운다.

## 모바일 양도증 관리

- 관리자 모바일 원본서류 페이지에 `양도증` 상태 선택을 추가한다.
- 선택값은 `자동`, `O`, `X`로 제공한다.
- `자동`은 로컬 파일 스캔 결과인 `cargo_cards.doc_transfer_received`를 사용한다.
- `O` 또는 `X`는 `cargo_original_docs.transfer_received_override`에 저장하고 자동 스캔보다 우선 표시한다.
- 수동값은 로컬 동기화가 `cargo_cards`를 갱신해도 유지된다.

## 원본 수령 확인 메일 추가 수신인

- 데스크톱 대시보드와 관리자 모바일 페이지의 H/C 원본서류 수령 메일 모달에 `추가 수신인` 입력란을 추가한다.
- 기본 수신인 `dmswk@hyundaicorp.com`, `ye25@hyundaicorp.com`은 항상 유지한다.
- 추가 수신인은 쉼표, 세미콜론 또는 줄바꿈으로 여러 주소를 입력할 수 있다.
- API에서 이메일 형식을 검증하고 중복 주소를 제거한 뒤 기본 수신인과 함께 To로 발송한다.
- 기존 참조인 `jsh@aincustoms.com`, `jhcho@aincustoms.com`, `bill@aincustoms.com`은 그대로 유지한다.
- 추가 수신인이 없으면 현재 메일 발송 방식과 동일하게 동작한다.

## 오류 처리 및 보안

- 로그인 세션과 B/L 소유권 검증은 기존 `cargo-quota` 검증을 그대로 사용한다.
- 화주는 자신의 계정 B/L만 수정할 수 있고 관리자는 전체 B/L을 수정할 수 있다.
- 날짜는 빈 값 또는 `YYYY-MM-DD`만 허용한다.
- 메일 발송에 필요한 SMTP 자격정보는 기존 Vercel 환경변수를 재사용하고 브라우저에 노출하지 않는다.
- 추가 수신인 검증은 서버에서 수행하며 잘못된 주소가 하나라도 있으면 발송하지 않고 정확한 오류를 반환한다.

## 테스트

- 상단 중복 메뉴가 렌더링되지 않는지 검사한다.
- 진행현황에 반입예정일 열과 편집 진입점이 있는지 검사한다.
- 화주가 실제 값을 변경할 때만 메일 함수가 호출되는지 검사한다.
- 관리자 수정, 동일 값 저장, 다른 수기 운송정보만 변경한 경우에는 메일이 호출되지 않는지 검사한다.
- 메일 실패 시 저장 성공 응답과 메일 실패 상태가 함께 반환되는지 검사한다.
- 진행현황 달력에 반입예정일 일정이 포함되는지 검사한다.
- 원본 X 클릭은 수령일 입력을 요구하고 O로 저장되는지 검사한다.
- 원본 O 클릭은 확인 후 X로 저장되며 수령일 팝업이 열리지 않는지 검사한다.
- 모바일 양도증 자동/O/X 상태가 API 응답과 저장 payload에 반영되는지 검사한다.
- 고정 수신인이 유지되고 유효한 추가 수신인이 To에 합쳐지는지 검사한다.
- 잘못된 추가 수신인은 서버에서 거부되는지 검사한다.
