# 서류전달 상태 설계

## 목표

BL별 진행현황의 `진행상태` 오른쪽에 `서류전달` 열을 추가한다. 관리자는 각 화물의 `삼현전달`과 `창고전달` 상태를 O/X로 관리하고, 화주 및 게스트 계정은 저장된 상태를 조회할 수 있다.

## 화면

- `진행상태`와 `동물검역` 사이에 `서류전달` 열을 배치한다.
- 한 셀에 `삼현`, `창고` 라벨과 작은 O/X 토글을 나란히 표시한다.
- 관리자는 토글을 클릭하면 변경 확인창을 본다. 확인하면 즉시 저장하고 취소하면 기존 상태를 유지한다.
- 화주 및 게스트 화면은 같은 O/X 표시를 제공하지만 클릭할 수 없다.
- 값이 없는 기존 화물은 두 상태 모두 X로 표시한다.

## 데이터

`public.cargo_card_user_inputs`에 다음 boolean 컬럼을 추가한다.

- `docs_delivered_samhyeon`
- `docs_delivered_warehouse`

두 컬럼의 기본값은 `false`다. 별도 이력이나 전달일은 이번 범위에 포함하지 않는다.

`public.shipper_accounts`에는 다음 계정 분류 컬럼을 추가한다.

- `account_category`: `shipper` 또는 `destination`

기존 계정은 `shipper`를 기본값으로 사용하고 CTF 계정은 `destination`으로 지정한다. 관리자 설정 화면에서 화주 계정의 계정 분류를 `화주/납품처`로 선택할 수 있게 해 이후 납품처 계정 추가에도 하드코딩 없이 대응한다.

## 저장 흐름

- 새 API 파일을 만들지 않고 기존 `/api/cargo-quota`의 `admin_status` 동작을 확장한다.
- 서버는 관리자 권한을 확인한 뒤 요청된 boolean 값만 저장한다.
- 화면은 저장 성공 응답을 받은 후 데이터를 다시 조회해 서버 상태와 일치시킨다.
- 오류가 발생하면 기존 화면값을 유지하고 오류 메시지를 표시한다.

## 연결 계정 일관성

같은 원본 화물은 `정규화된 B/L 번호 + 정확한 폴더명`으로 식별한다. HCH와 CTF처럼 같은 화물이 여러 계정에 복제되어 있으면 두 전달 상태를 연결된 모든 계정의 입력 행에 함께 저장한다. 조회 시에도 연결 화물의 값을 병합해 계정에 따라 상태가 달라지지 않게 한다.

## 입력구분 표시

운송정보 입력내역 팝업의 `입력구분`은 저장 당시 계정 정보를 기준으로 다음과 같이 표시한다.

- 관리자 계정: `관리자(AIN)`
- `account_category = shipper`: `화주`
- `account_category = destination`: `납품처`

CTF 계정에서 저장한 운송정보는 `납품처`로 표시한다. 팝업에는 기존과 같이 입력자 계정과 입력일시도 유지한다. 입력구분은 저장 당시 값으로 보존해 나중에 계정 분류가 바뀌어도 과거 입력의 출처가 변하지 않게 한다.

## 로컬 연동

- 로컬 동기화는 두 컬럼을 내려받아 로컬 카드 데이터에 포함한다.
- 홈페이지 원본, 로컬 통합 사본, Vercel 패키지 사본은 동일한 표시 및 저장 규칙을 사용한다.
- 로컬 관리 화면에서도 관리자는 동일한 두 상태를 변경할 수 있다.

## 테스트

- 신규 컬럼과 기본값을 검증하는 마이그레이션 테스트
- `admin_status`가 두 값을 정규화해 저장하는 API 테스트
- 연결 계정 모두에 상태가 전파되는 테스트
- 관리자만 토글 가능하고 화주·게스트는 읽기 전용인 화면 테스트
- `진행상태` 바로 오른쪽에 `서류전달` 열이 배치되는 마크업 테스트
- 관리자·화주·납품처 입력구분과 CTF 계정 분류를 검증하는 테스트
- 홈페이지·로컬 사본 일치 및 전체 회귀 테스트
